# 系统设计评估与演进方案

本文档回答一个核心问题:当前系统设计是否完善。结论是:功能实现已经比较丰富,局部模块设计也清楚,但顶层系统设计还不完善。主要问题不是缺少功能,而是缺少一份统一说明来约束后续演进:领域边界、数据生命周期、数据源适配、任务一致性、缓存失效和能力门控需要被放到同一张图里。

## 1. 当前结论

当前系统处于"实现领先于架构文档"的状态。

已经比较成熟的部分:

- 产品方向已经从通用行情面板收敛到"盘后复盘、策略跟踪、观察池、监控提醒"。
- 后端形成了清晰的本地数据技术栈:Parquet 存储、DuckDB 冷查询、Polars 热路径计算与缓存。
- TickFlow / AkShare 双数据源方向已经起步,且 AkShare 模式被限制为手动盘后同步,符合免费数据源稳定性边界。
- 能力探测、SSE 推送、策略缓存、监控规则、回测任务等关键模块已经具备实际可用的骨架。
- "关注范围 sync" 和 "全市场 sync" 的区分已经出现,这对个人工作台是正确方向。

尚不完善的部分:

- 缺少顶层领域模型:复盘、观察池、策略生命周期、监控、回测之间的主从关系还没有成为代码层面的核心模型。
- 数据源边界还不彻底:已经有 `datasource/`,但部分服务仍直接依赖 TickFlow 语义或能力档位。
- 数据契约没有集中定义:Parquet 分区、字段口径、原始价/复权价、实时覆盖、扩展数据 join 规则分散在代码和注释里。
- 写入与缓存一致性需要更明确:盘后管道、实时行情、五档盘口、扩展数据都可能影响页面查询,但缺少一张统一的"谁写、谁刷新、谁失效"表。
- 能力门控仍有历史遗留:部分判断应完全基于 capability/provider,避免 label/tier 在新数据源下产生误判。
- 缺少系统级验收矩阵:目前测试集中在回测正确性,但缺少 provider、pipeline、cache、SSE、降级行为的端到端验收清单。

所以,下一阶段不建议继续优先加大功能面,而应该先补齐系统设计边界。

## 2. 目标系统定位

系统应被定义为:

> 面向个人投资者和低频量化爱好者的本地优先 A 股研究工作台,以盘后复盘、策略跟踪、观察池和计划执行提醒为核心闭环。

不是:

- 通用行情终端
- 资讯门户
- 高频盯盘工具
- AI 自动荐股系统
- 自动交易执行系统

这个定位决定了后续架构取舍:

- 默认同步关注范围,全市场同步作为显式重任务。
- 实时行情是增强能力,不是核心依赖。
- AI 只做策略生成、复盘整理和假设质疑,不做买卖结论。
- 数据源可以降级,但本地研究闭环不能崩。

## 3. 目标架构

```text
React SPA
  -> API client / TanStack Query / SSE invalidation
  -> FastAPI routers
  -> application services
     -> review / observation / strategy / monitor / backtest workflows
     -> pipeline jobs / scheduler / progress store
  -> domain services
     -> strategy engine
     -> indicator pipeline
     -> monitor rule engine
     -> backtest engine
  -> infrastructure
     -> datasource adapters: TickFlow / AkShare / future providers
     -> repository: Parquet + DuckDB views + Polars caches
     -> local user data: preferences / secrets / alerts / strategy configs
```

关键原则:

- 前端只理解产品能力和 API 契约,不理解真实数据源细节。
- 业务服务只依赖标准化后的本地数据和 capability,不直接散落 provider 特判。
- 数据源适配层只负责拉取、规范化、容错,不承载策略和页面语义。
- repository 是唯一存储入口,负责分区写入、视图刷新和热缓存读取。
- 长任务都进入 job/progress 体系,避免页面直接等待重任务完成。

## 4. 模块边界

| 模块 | 当前职责 | 目标边界 |
| :--- | :--- | :--- |
| `config.py` | 读取环境变量、桌面版路径、用户覆盖配置 | 只提供配置,不承载业务判断 |
| `datasource/` | AkShare 规范化工具和 provider registry | 所有外部数据源统一入口,输出标准 DataFrame |
| `tickflow/` | TickFlow SDK、能力探测、仓储 | TickFlow provider 适配和 capability 来源,不应外溢到业务层 |
| `repository.py` | Parquet 写入、DuckDB 视图、Polars 缓存 | 唯一存储访问层,沉淀数据生命周期契约 |
| `daily_pipeline.py` | 盘后同步、指标计算、调度注册 | 编排任务,不内联具体 provider 细节 |
| `quote_service.py` | 实时行情、盘中 enriched、SSE 事件 | TickFlow 实时增强旁路,AkShare 模式必须自然禁用 |
| `indicators/pipeline.py` | 技术指标、信号、enriched 生成 | 纯计算层,避免读写和 provider 依赖 |
| `strategy/` | 策略定义、运行、AI 生成、监控规则 | 策略生命周期应继续上移成显式领域模型 |
| `backtest/` | 回测引擎和服务 | 允许 pandas/vectorbt 边界集中在回测层 |
| `ext_data/` | 扩展数据配置、上传、HTTP 拉取 | 作为本地数据扩展入口,必须遵守 schema/version 契约 |
| `frontend/src/lib/api.ts` | 前端统一 API client 和类型 | 与后端响应保持同步,不要让页面重复猜 provider |

## 5. 数据生命周期

### 5.1 盘后路径

```text
provider fetch
  -> normalize symbol/date/OHLCV
  -> write kline_daily / instruments / index tables
  -> compute enriched
  -> write kline_daily_enriched
  -> refresh DuckDB views
  -> refresh Polars cache
  -> invalidate data/status and query caches
  -> pages read local data
```

设计要求:

- provider 失败不能清空旧数据。
- 写入必须以本地标准字段为准。
- `kline_daily_enriched` 磁盘表保持窄表,完整指标可在内存或查询时计算。
- 关注范围和全市场范围必须在 job result 中显式体现。

### 5.2 盘中路径

```text
TickFlow realtime quote
  -> overwrite today's kline_daily
  -> incremental enriched calculation
  -> update repository hot cache
  -> evaluate monitor rules
  -> write alerts
  -> push SSE
```

设计要求:

- 盘中路径是增强能力,不能成为 AkShare / Free 模式的隐含依赖。
- 实时写入当天分区时,需要和盘后管道保持明确的覆盖规则。
- SSE 只传事件和失效信号,页面仍应通过 API 读取权威数据。

### 5.3 扩展数据路径

```text
CSV / Excel / JSON / HTTP pull
  -> schema discovery
  -> symbol normalization
  -> ext parquet
  -> DuckDB view
  -> API joins selected ext columns
```

设计要求:

- 扩展数据不应污染策略结果缓存的核心 key。
- 扩展字段 join 只能发生在展示或明确查询阶段。
- 每个扩展表需要记录 schema、mode、同步时间和字段口径。

## 6. 一致性与缓存设计

当前缓存分为:

- DuckDB 视图:冷查询、统计、元数据、扩展数据。
- Polars cache:最新 enriched、instruments、指数维表、盘中聚合状态。
- 文件缓存:策略结果、capabilities、preferences、alerts。
- 前端 query cache:TanStack Query。

需要补齐的统一规则:

| 事件 | 写入者 | 必须刷新 | 必须失效 |
| :--- | :--- | :--- | :--- |
| instruments 写入 | instrument sync / akshare sync | instruments view, instruments cache | data status, symbol search |
| daily 写入 | pipeline / quote service | kline_daily view | data status, K 线查询 |
| enriched 写入 | pipeline / quote service | kline_enriched view, enriched cache | screener, watchlist, overview, monitor |
| index 写入 | index sync | index views/cache | dashboard indices |
| ext data 写入 | ext_data service / pull scheduler | ext views | affected ext pages and joined tables |
| monitor alert 写入 | monitor engine | alert store | alerts, unread badge |
| strategy config change | strategy API | strategy engine/cache | screener cached result, backtest params |

后续实现应尽量把这些规则集中到一个 cache/event registry,减少各 API 手工散落调用。

## 7. 能力与数据源门控

目标判断顺序:

1. 当前 provider 是什么。
2. 当前 provider 暴露哪些 capability。
3. 当前页面/接口需要哪些 capability。
4. 不满足时给出降级 UI 或明确 403。

不推荐:

- 用 `tier_label` 直接判断功能是否可用。
- 把 `AkShare` 当成一种 TickFlow 档位。
- 前端通过字符串猜测后端真实能力。

推荐契约:

```text
GET /api/capabilities
  provider: tickflow | akshare
  label: human readable label
  capabilities: normalized capability map
```

所有页面、按钮、后台任务都应以 capability map 为准。

## 8. 优先级路线

### Phase A: 设计基线

目标:把系统边界说清楚。

- 保持本文档作为系统设计总入口。
- 给数据表、provider、capability、cache/event 增加集中索引:
  - [数据契约与缓存刷新.md](./数据契约与缓存刷新.md)
  - [数据源与能力门控契约.md](./数据源与能力门控契约.md)
  - [总计划与投资进度设计.md](./总计划与投资进度设计.md)
  - [开发路书.md](./开发路书.md)
- 在新增功能前检查是否服务"长期规划、复盘、观察池、策略、监控、回测、数据"主线。

验收:

- README 能链接到系统设计。
- 数据契约与 provider/capability 契约能被后续任务直接引用。
- 新任务可以引用本文档判断是否越界。
- 关键模块都有明确 owner 和上下游。

### Phase B: Provider 边界硬化

目标:让 TickFlow / AkShare 都只作为数据源适配存在。

- 为 TickFlow 补一个 datasource adapter 或 facade。
- 同步服务优先依赖 provider 接口,减少直接 SDK 调用。
- 所有接口改为 capability/provider 门控。
- AkShare 模式下所有 TickFlow 专属操作自然灰显或 403。

验收:

- 切换 `DATA_PROVIDER=akshare` 不会触发 TickFlow 网络请求。
- 免费数据源模式能完成关注范围日线同步、指标、选股、K 线、基础回测。
- TickFlow 模式行为不退化。

### Phase C: 长期规划与总计划 MVP

目标:把"10 年做到 150 万"这类长期目标从静态 Markdown 变成可维护规划。

- 创建总计划一级页面。
- 支持目标金额、目标日期、初始本金、基准月投。
- 支持月投入情景、年度里程碑、阶段执行计划、风险规则。
- 支持持仓、现金、净投入和年度复盘记录。
- summary 返回计划资产、目标完成度、里程碑状态、当前所需年化和静默健康度。

验收:

- 用户能载入或创建长期目标规划。
- 用户能看到不同月投入对应的目标难度。
- 用户能看到当前进度相对年度里程碑的领先、正常或落后状态。
- 用户能录入持仓、净投入和年度复盘,刷新后仍存在。

### Phase D: 观察池与复盘领域模型

目标:让产品主线进入数据模型,而不只是导航文案。

- 将普通 watchlist 升级为观察池模型。
- 记录来源、理由、交易假设、触发价、失效价、计划、状态、复盘备注。
- 策略命中可以一键加入观察池。
- 复盘页读取观察池和策略结果生成明日计划。

验收:

- 用户能从策略命中到观察池,再到次日复盘形成闭环。
- 监控规则可以引用观察项的触发/失效条件。

### Phase E: 任务与缓存一致性

目标:减少"数据写了但页面没刷新"或"缓存旧数据误导"的问题。

- 抽出 cache/event registry。
- job result 明确写入了哪些表、哪些 scope、哪些失败。
- 前端 SSE invalidation 和后端 cache invalidation 使用同一语义表。
- 写 parquet 时形成原子写入策略,避免半写分区被读取。

验收:

- 每个写入路径都有测试或手工验收说明。
- 刷新、切页、重连后的页面状态一致。

### Phase F: 系统级测试矩阵

目标:从功能可用走向系统可信。

至少覆盖:

- TickFlow none/free/starter+ capability 行为。
- AkShare provider 手动同步和失败保留旧数据。
- focus sync 与 full-market sync。
- pipeline job 重入、进度、失败结果。
- enriched cache 刷新和前端 SSE invalidation。
- 回测数据边界和 pandas/vectorbt 边界。
- 扩展数据 schema 发现、写入、join。

## 9. 新功能设计检查表

新增功能前先问:

1. 它服务哪个主工作流:长期规划、复盘、观察池、策略、回测、监控、数据?
2. 它依赖哪些 capability? 不满足时如何降级?
3. 它读写哪些本地表? 谁负责刷新视图和缓存?
4. 它是否引入新的 provider 直接调用? 如果是,为什么不能放进 datasource?
5. 它是否会改变 enriched 字段口径? 是否需要 schema/version 说明?
6. 它是否会影响 SSE 刷新或前端 query cache?
7. 它是否需要加入系统级测试矩阵?

如果多数问题答不上来,应先补设计再写代码。

## 10. 近期建议

短期最值得做的不是继续堆新页面,而是按以下顺序推进:

1. 先把 provider/capability/data lifecycle 三个契约补硬。
2. 把长期规划/总计划模型落地,让 10 年目标从静态文档变成长期投资导航层。
3. 再把观察池模型落地,让"复盘工作台"有真实业务对象。
4. 最后整理 cache/event registry 和测试矩阵,提升长期维护确定性。

这样项目会从"功能很全的股票面板"进一步变成"有边界、有闭环、能持续演进的个人交易研究系统"。

## 11. 当前实现核对

这次按代码事实抽样核对后,可以把完整度拆成四类:

| 方面 | 当前状态 | 设计判断 |
| :--- | :--- | :--- |
| 存储与查询 | `DataStore` / `KlineRepository` 已经统一 Parquet、DuckDB view 和 Polars 热缓存 | 方向正确,但写入原子性、局部刷新和失效语义仍靠各调用点自觉维护 |
| 数据源与能力 | `CapabilitySet` 已成为主要能力来源,AkShare 已作为 provider 独立返回静态 capability | 基线已经有了,但仍有 `tier_label()` / `tierRank()` 参与真实门控,需要收口到 capability metadata |
| 长任务 | pipeline job store、进度、复用 running job、AkShare 手动管道都已落地 | 可用性不错,但 job result 还没有明确声明"写了哪些表、影响哪些 cache/query" |
| 产品领域 | 前端已经把 Dashboard 命名为复盘、Watchlist 命名为观察池 | 语义先行,模型滞后;后端仍是 `symbol + added_at + note`,还没有观察项、假设、计划、状态、复盘记录 |

所以当前设计不是"不完善到需要推倒",而是"已经进入需要硬化边界的阶段"。继续堆功能会放大维护成本;先补系统设计是正确优先级。

## 12. 下一阶段设计落地边界

### 12.1 先做领域闭环,不要先做新页面

优先把 `watchlist` 升级为观察池领域对象:

```text
ObservationItem
  id
  symbol
  name
  source: manual | screener | alert | import
  source_ref: strategy_id / alert_id / ext_ref
  thesis: 观察理由 / 交易假设
  trigger_price
  invalid_price
  target_price
  status: tracking | triggered | invalidated | archived
  priority
  tags
  created_at
  updated_at
  review_notes[]
```

这个模型落地后,复盘、策略命中、监控规则、次日计划才有共同对象。否则"观察池"只是自选股页面的文案升级。

### 12.2 再做 capability metadata,不要只替换字符串

下一步不要把 `tier_label()` 机械替换为 `provider` 字符串。应补充 capability 的限制表达:

```text
CapabilityLimits
  rpm
  batch
  subscribe
  min_interval
  max_interval
  max_history_days
  realtime_allowed
```

随后让实时行情开关、五档轮询间隔、分钟 K 历史扩展、前端灰显都从这份 metadata 派生。这样 TickFlow、AkShare 和未来 provider 都能用同一套语义。

### 12.3 建一个写入事件表,不要让缓存失效散落

建议新增一个系统级事件表,先文档化,再实现:

```text
DataMutation
  event: instruments_written | daily_written | enriched_written | ext_written | alerts_written
  tables: [...]
  duckdb_views: [...]
  polars_caches: [...]
  frontend_queries: [...]
  sse_events: [...]
```

pipeline、quote service、ext data、monitor alert 都上报同一种事件。后端 cache refresh 和前端 query invalidation 再从这张表消费,减少"数据已写但页面没刷"的隐性 bug。

### 12.4 系统设计验收标准

这一阶段可以认为"系统设计基本完善"的标准:

1. 任一新功能能明确归属到复盘、观察池、策略、监控、回测、数据六条主线之一。
2. 任一接口能说清楚依赖哪些 capability,不满足时返回什么降级行为。
3. 任一写入路径能说清楚写入哪些表、刷新哪些 DuckDB view、刷新哪些 Polars cache、失效哪些前端 query。
4. 观察池不再只是 symbol 列表,而是能承载策略来源、交易假设、触发/失效条件和复盘记录。
5. AkShare 模式不会触发 TickFlow 网络请求;TickFlow none/free/starter+ 和 AkShare 都有系统级验收用例。
6. `tier_label` 只用于展示,不再承担真实业务门控。

满足以上标准后,再继续扩展行情、板块、AI 分析或交易执行,系统会稳很多。
