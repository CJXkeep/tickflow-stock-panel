# 数据源与能力门控契约

本文档定义 provider 与 capability 的系统级契约。目标是让 TickFlow、AkShare 和未来数据源都通过同一套能力语义接入,避免业务代码依赖档位字符串或 provider 私有细节。

## 1. 当前 Provider

| Provider | 定位 | 当前支持 | 明确不支持 |
| :--- | :--- | :--- | :--- |
| `tickflow` | 默认商业数据源 | 能力探测、日 K、指数、实时行情、分钟 K、除权因子、五档、财务等,按订阅 capability 决定 | 无能力时必须降级 |
| `akshare` | 免费盘后数据源 | 手动同步股票维表、关注范围或全市场日 K、核心指数日 K、基础 enriched、选股、K 线、基础回测 | 自动联网调度、实时行情、分钟 K、五档、财务、除权因子 |

AkShare 是 provider,不是 TickFlow 档位。系统不得把 `AkShare` 当作 `Free` / `Starter` / `Pro` 一类 tier 处理。

## 2. 能力来源

后端对外统一暴露:

```text
GET /api/capabilities
  provider: tickflow | akshare
  label: human readable label
  capabilities: Record<capability, limits>
```

当前 capability 枚举:

```text
quote.by_symbol
quote.batch
quote.pool
kline.daily.by_symbol
kline.daily.batch
kline.minute.by_symbol
kline.minute.batch
intraday
intraday.batch
depth5
depth5.batch
websocket
financial
adj_factor
```

TickFlow 模式:

- 有 Key 时走真实探测。
- 无 Key 或无效 Key 时落到 none/free 能力。
- label 只用于展示,业务判断必须依赖 `CapabilitySet`。

AkShare 模式:

- 返回静态能力集。
- 当前只暴露日线相关能力。
- 不发起 TickFlow 探测。
- 调度器不自动注册 provider 网络同步。

## 3. 门控规则

后端接口规则:

- 需要能力的接口使用 `capset.require(Cap.X)` 或等价 provider-aware 判断。
- provider 专属接口必须先判断 `settings.provider_is_akshare` 或 registry。
- 不满足能力时返回明确 403 或受控降级结果。
- 不得只用 `tier_label()` 判断功能可用。

前端规则:

- 页面从 `/api/capabilities` 和 `/api/settings` 获取 provider 与能力。
- 按 capability 控制按钮可用性和提示。
- 不通过字符串猜测订阅档位来决定真实后端能力。
- 灰显能力必须说明是数据源不支持还是当前订阅不支持。

服务层规则:

- pipeline 编排可以按 provider 分支,但具体拉取逻辑应留在 provider sync service。
- 策略、回测、监控应优先读取本地标准化数据,不直接关心 provider。
- realtime、depth、financial 等增强能力必须是旁路能力,不能成为基础日线闭环的隐含依赖。

## 4. Provider 行为矩阵

| 行为 | TickFlow | AkShare |
| :--- | :--- | :--- |
| 启动时能力探测 | 是,可读缓存 | 静态生成 |
| 启动时自动联网同步 | 允许按调度配置 | 不允许 |
| 手动盘后 pipeline | 支持 focus / market | 支持 focus / market |
| 关注范围 sync | 支持 | 支持 |
| 全市场 sync | 订阅能力允许时支持 | 支持但较慢,用户显式触发 |
| 实时行情 | starter+ 能力 | 不支持 |
| 分钟 K | capability 决定 | 不支持 |
| 五档盘口 | capability 决定 | 不支持 |
| 财务数据 | expert 能力 | 暂不支持 |
| 除权因子 | starter+ 能力 | 暂不支持,AkShare 日线口径需单独说明 |
| 指数日 K | 支持 | 支持核心指数 |

## 5. 禁止的依赖方向

禁止:

```text
strategy/backtest/monitor -> TickFlow SDK
frontend page -> provider private field
business service -> tier label string
AkShare provider -> TickFlow capability cache
datasource adapter -> FastAPI request/app.state
```

允许:

```text
pipeline -> provider sync service
provider sync service -> datasource adapter
business service -> repository
business service -> CapabilitySet
frontend -> documented API response
```

## 6. 新增 Provider 检查表

新增 provider 时必须完成:

1. 在配置中加入 provider 名称,并提供规范化 label。
2. 实现 symbol/date/OHLCV 规范化。
3. 定义静态或探测型 capability set。
4. 明确哪些 pipeline stage 支持、降级或禁用。
5. 不改变 repository 的标准写入契约。
6. 不让前端页面直接读取 provider 私有字段。
7. 补充数据口径说明,尤其是复权、成交量单位、换手率和指数。
8. 给失败保留旧数据和部分失败结果提供策略。

## 7. 近期硬化任务

建议按以下顺序做:

1. 将所有 label/tier 功能判断改为 capability/provider 判断。
2. 为 TickFlow 增加 datasource facade,让同步服务不直接散落 SDK 调用。
3. 建立 provider stage matrix,供数据页和 pipeline 共享。
4. 为 AkShare 模式补齐所有 TickFlow 专属接口的明确降级响应。
5. 在系统级测试矩阵中覆盖 `tickflow none/free/starter+` 和 `akshare`。

## 8. 已发现设计债

以下位置仍带有历史档位字符串判断。它们不一定是当前 bug,但应在 provider/capability 硬化阶段处理:

| 位置 | 当前行为 | 建议演进 |
| :--- | :--- | :--- |
| `backend/app/api/kline.py` 的 `extend_minute_history` | `month` 单位通过 `tier_label()` 判断 Expert | 增加显式能力,如 `kline.minute.long_history`,或在 capability limits 中表达最大回溯范围 |
| `backend/app/services/depth_service.py` 的 `_compute_interval` | 轮询间隔范围由 `tier_label()` 映射 | 将可配置间隔范围放进 depth capability metadata,或由 rpm/batch 推导 |
| `frontend/src/lib/capability-labels.tsx` | `tierRank()` 仍用于部分 UI 门控 | 后续只保留展示用途;真实可用性由 `/api/capabilities` 派生 |
| `frontend/src/components/data/DepthConfigCard.tsx` | 前端用档位 label 决定轮询范围文案 | 后端返回可配置范围,前端只展示服务端契约 |

处理这些债务时不要只把 `tier_label()` 替换成另一个隐含字段。正确做法是先补充 capability 表达能力,再让前后端共同消费这个能力。
