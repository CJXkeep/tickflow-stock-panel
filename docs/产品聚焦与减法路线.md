# 产品聚焦与减法路线

本文档用于重新定义本项目的产品方向。核心判断是:不要继续复制通用炒股软件,而是把项目收窄为个人交易研究、策略跟踪和盘后复盘工作台。

## 1. 背景

当前项目已经具备行情、K 线、选股、回测、监控、行业概念、数据同步等能力,但其中不少能力会天然落入同花顺、通达信、东方财富等成熟软件的覆盖范围。

这些软件在以下领域已经很强:

- 实时行情
- 盘口和逐笔
- 通用 K 线交互
- 资讯和公告入口
- 热门榜单
- 板块涨跌展示
- 通用指标叠加

如果继续沿着大而全的方向做,本项目很容易变成一个低配版行情终端。更合理的路线是避开通用行情展示,专注成熟软件做得较弱、但个人交易者真正需要沉淀的环节:

- 盘后复盘
- 策略生命周期管理
- 明日观察池
- 个股跟踪理由
- 策略语义提醒
- 交易计划和结果复盘
- AI 辅助总结与质疑

## 2. 产品定位

一句话定位:

> 面向个人散户和低频量化爱好者的盘后复盘、策略跟踪与明日交易计划工作台。

项目不追求成为全功能股票软件,而是成为用户每天收盘后和开盘前反复使用的个人决策系统。

目标工作流:

1. 收盘后同步数据
2. 自动复盘市场状态
3. 评估策略当日表现
4. 生成明日观察池
5. 盘中只提醒关键语义变化
6. 收盘后记录观察结果
7. 持续更新策略评分和交易规则

## 3. 目标用户

优先服务以下用户:

- 有固定复盘习惯的个人投资者
- 使用技术面、情绪面或题材轮动做低频交易的用户
- 有自定义策略、但缺少跟踪系统的用户
- 想把选股、回测、观察、复盘串起来的用户
- 能接受盘后数据为主、不强依赖毫秒级实时行情的用户

暂不优先服务:

- 高频交易用户
- 强盘口交易用户
- 只想看实时行情和资讯的用户
- 只需要一个更漂亮 K 线软件的用户
- 完全依赖 AI 荐股的用户

## 4. 产品原则

### 4.1 数据展示必须服务决策

页面不要只是展示已有数据,必须回答一个具体问题。例如:

- 今天市场强弱如何?
- 哪些策略有效?
- 哪些候选股值得明天观察?
- 哪些观察股已经失效?
- 哪些规则需要调整?

如果一个模块只能展示数据,但不能推进用户决策,就应该降级、隐藏或删除。

### 4.2 策略少而深

内置策略数量不是核心价值。更重要的是让少数策略被完整跟踪:

- 策略假设
- 参数版本
- 历史回测
- 每日命中
- 次日表现
- 持仓/观察结果
- 失效原因
- 调整记录

20 个浅策略不如 5 个能被持续复盘的策略。

### 4.3 观察池是核心资产

普通自选股只记录“我关注它”。本项目的观察池应该记录“我为什么关注它,什么情况下继续看,什么情况下放弃”。

每个观察项至少应该能沉淀:

- 来源策略
- 加入理由
- 关键价位
- 风险点
- 失效条件
- 计划动作
- 复盘备注
- 结果状态

### 4.4 提醒要有语义

普通价格提醒价值有限。提醒应该尽量表达交易语义:

- 观察池股票触发买点
- 昨日强势股低开后翻红
- 策略命中股次日走弱
- 板块主线退潮
- 候选股突破关键价但市场环境偏弱

提醒不是为了制造噪音,而是减少用户盯盘时间。

### 4.5 AI 不做荐股,只做复盘助手

AI 模块不应该直接给“买什么”。更适合承担:

- 总结市场
- 解释策略表现
- 质疑用户假设
- 归纳亏损交易共性
- 从复盘笔记中提炼规则
- 把候选池整理成明日计划

## 5. 专精方向

### 5.1 盘后复盘中心

盘后复盘中心是未来首页的主要候选。它不应是行情大屏,而应是一张每日研究报告。

核心问题:

- 今天市场处于进攻、防守还是混沌状态?
- 指数、成交额、涨跌家数、连板高度是否支持交易?
- 哪些行业或概念在加强?
- 哪些方向在退潮?
- 昨天观察池表现如何?
- 今天哪些策略命中新标的?
- 明天应该关注哪些股票?

建议模块:

- 市场温度
- 主线强弱
- 策略命中概览
- 观察池结果复盘
- 明日重点观察
- 风险提示
- AI 复盘摘要

### 5.2 策略生命周期

策略页从“策略列表”升级为“策略运营台”。

每个策略需要有生命周期状态:

- 草稿
- 回测中
- 观察中
- 实盘跟踪
- 暂停
- 废弃

每个策略需要记录:

- 策略假设
- 适用市场环境
- 不适用环境
- 参数版本
- 回测摘要
- 最近 N 次命中表现
- 最近失效案例
- 用户备注

策略卡片不只显示“今天命中几只”,还应显示:

- 近 20 日命中后次日胜率
- 平均次日收益
- 最大回撤
- 最近是否降温
- 是否适合明天继续使用

### 5.3 明日观察池

观察池是连接策略和真实决策的核心页面。

观察项来源:

- 策略命中
- 用户手动加入
- AI 复盘建议
- 监控触发
- 回测案例收藏

观察项字段:

| 字段 | 说明 |
| :--- | :--- |
| symbol | 股票代码 |
| name | 股票名称 |
| source | 来源,如策略/手动/AI/监控 |
| reason | 加入理由 |
| thesis | 交易假设 |
| trigger_price | 关键触发价 |
| invalid_price | 失效价 |
| risk_note | 风险点 |
| plan | 明日计划 |
| status | 观察中/触发/失效/完成 |
| review_note | 复盘备注 |

观察池应该支持每日结转:

- 昨日观察池表现
- 自动标记触发/失效候选
- 用户确认保留或移除
- 生成今日观察池

### 5.4 策略语义监控

监控中心从“规则集合”升级为“计划执行提醒”。

优先支持:

- 观察池触发价提醒
- 观察池失效价提醒
- 策略命中后的次日跟踪提醒
- 板块强弱变化提醒
- 市场环境变化提醒
- 用户自定义信号提醒

提醒内容应该包含:

- 为什么提醒
- 来自哪个策略或观察计划
- 当前触发了什么条件
- 是否和原计划冲突
- 建议用户检查什么

### 5.5 AI 复盘助手

AI 入口不做独立聊天页优先,而是嵌入复盘和策略工作流。

优先能力:

- 生成每日复盘草稿
- 总结策略表现
- 解释观察池结果
- 找出亏损交易共性
- 帮用户把自然语言想法转成策略规则
- 根据复盘备注提炼可执行检查项

明确边界:

- 不输出确定性买卖建议
- 不承诺预测涨跌
- 不替代用户决策
- 不在缺少数据证据时编造结论

## 6. 减法方向

### 6.1 降级通用行情页

以下能力可以保留为辅助入口,但不应占据产品主线:

- 通用指数行情
- 普通 K 线浏览
- 普通涨跌榜
- 普通板块列表
- 常规指标叠加

处理方式:

- 从首页降级到详情页或辅助入口
- 默认只在个股、策略、观察池上下文中打开
- 不再围绕它们新增大量通用功能

### 6.2 收敛实时行情执念

项目不应优先追求完整实时行情终端。

保留:

- 关键观察股提醒
- 策略语义触发
- 低频价格/涨跌变化

暂缓:

- 分时逐笔
- 全市场实时扫描
- 五档盘口深度
- 高频刷新看板

### 6.3 减少页面平铺

未来导航不宜继续按数据类型平铺,如行情、指数、行业、概念、财务、监控、策略、回测全部同级。

建议按用户工作流重组:

- 复盘
- 观察池
- 策略
- 回测
- 监控
- 数据
- 设置

行业、概念、个股分析、K 线等作为上下文页面进入,不作为第一层主线入口。

### 6.4 减少纯指标堆叠

指标应该转成结论。页面上尽量减少“指标大全”式展示。

从:

- MA5
- MA10
- RSI
- KDJ
- MACD
- 量比

转成:

- 趋势转强
- 缩量回踩
- 放量突破
- 超跌反弹
- 高位分歧
- 量价背离

### 6.5 暂缓大而全财务模块

财务数据可以作为策略因子和过滤条件存在,但暂不优先做成完整财务分析软件。

优先保留:

- 市值
- 流通市值
- 基础估值
- 财务风险过滤
- 策略需要的关键因子

暂缓:

- 完整三大报表分析
- 财务图表大屏
- 财报研报式页面

## 7. 推荐信息架构

### 7.1 第一层导航

建议未来主导航收敛为:

| 页面 | 定位 |
| :--- | :--- |
| 复盘 | 每日市场总结、策略表现、明日计划 |
| 观察池 | 明日候选、跟踪理由、触发和失效 |
| 策略 | 策略生命周期、命中、评分、调参 |
| 回测 | 验证策略假设和参数 |
| 监控 | 执行观察计划和策略语义提醒 |
| 数据 | 同步、数据健康、扩展数据 |
| 设置 | 数据源、Key、偏好、系统设置 |

### 7.2 二级上下文页面

以下页面作为上下文能力,不优先放在第一层:

- 个股详情
- K 线
- 行业分析
- 概念分析
- 指数详情
- 财务详情
- 扩展数据详情

入口来自:

- 观察池股票
- 策略命中结果
- 复盘报告
- 监控提醒
- 搜索

## 8. 阶段路线

### Phase 1: 文档和产品主线确认

目标:

- 确认产品定位
- 确认做什么和不做什么
- 确认未来导航和页面优先级

交付:

- 本文档
- README 中的定位更新
- 待办路线拆分

### Phase 2: 复盘首页原型

目标:

- 用已有数据生成可用的盘后复盘视图
- 暂不追求 AI 完整自动化

最小能力:

- 市场温度
- 今日策略命中
- 昨日观察池表现
- 明日候选池草稿
- 风险提示

### Phase 3: 观察池重构

目标:

- 把普通 watchlist 升级为带理由、计划、状态的观察池

最小能力:

- 策略命中一键加入观察池
- 手动编辑理由和计划
- 触发价/失效价
- 每日复盘状态
- 保留/移除/完成流转

### Phase 4: 策略评分和生命周期

目标:

- 让策略从静态列表变成可持续跟踪的系统

最小能力:

- 策略状态
- 最近命中表现
- 次日表现统计
- 参数版本备注
- 策略暂停/启用

### Phase 5: 语义提醒和 AI 复盘

目标:

- 把监控和 AI 接入真实工作流

最小能力:

- 观察池触发提醒
- 策略次日跟踪提醒
- AI 生成复盘草稿
- AI 总结策略表现
- AI 根据复盘备注提炼规则

## 9. 优先级判断标准

以后新增功能前,先问以下问题:

1. 它是否服务盘后复盘、观察池、策略跟踪或计划执行?
2. 它是否能沉淀用户自己的交易系统?
3. 它是否比通用炒股软件更贴近个人决策流程?
4. 它是否能减少用户盯盘和重复整理?
5. 它是否能形成每日可重复工作流?

如果答案大多是否,就应该延后。

## 10. 不做清单

短中期明确不优先:

- 对标同花顺/通达信的完整行情终端
- 完整盘口交易工具
- 逐笔成交分析
- 资讯聚合门户
- 全功能财务分析平台
- AI 自动荐股
- 大量无复盘闭环的内置策略
- 纯展示型大屏

## 11. 成功指标

产品不应只看页面数量和功能数量,更应该看使用闭环是否成立。

建议指标:

- 用户每周完成几次盘后复盘
- 每日观察池是否被更新
- 策略命中后是否有跟踪结果
- 观察项是否记录了理由和失效条件
- 监控提醒是否来自明确计划
- AI 摘要是否被用户编辑并保存
- 策略是否因为复盘结论发生迭代

## 12. 下一步建议

最适合先做的不是继续扩功能,而是做一个“复盘工作台”最小闭环:

1. 新增复盘页或重构 Dashboard 为复盘页
2. 增加观察池数据模型
3. 策略命中结果支持加入观察池
4. 观察池支持理由、计划、触发价、失效价
5. 每日同步后生成复盘摘要和明日观察池草稿

这条路线会让项目从“股票数据面板”变成“个人交易系统操作台”。
