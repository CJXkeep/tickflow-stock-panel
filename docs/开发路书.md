## 1. 总目标

项目下一阶段的目标不是继续扩页面,而是把系统从"功能丰富的股票面板"推进成:

> 本地优先的个人 A 股研究工作台,以总计划、盘后复盘、观察池、策略跟踪、监控提醒和回测验证形成闭环。

开发优先级遵循三条规则:

1. 先稳边界,再扩功能。
2. 先让核心领域对象落地,再做更多展示。
3. 任一写入路径必须说明数据表、缓存、前端失效和降级行为。

## 2. 当前基线

已经具备:

- React SPA + FastAPI + Parquet / DuckDB / Polars 本地数据栈。
- TickFlow / AkShare 双 provider 基线。
- 盘后同步、关注范围同步、策略筛选、基础回测、监控规则、SSE 刷新。
- 系统设计、数据契约、provider/capability 契约、总计划设计文档。
- 已有一份长期目标规划样例: [10年做到150万投资目标规划.md](./计划/10年做到150万投资目标规划.md)。

仍需补齐:

- `tier_label` / `tierRank` 等历史档位判断需要收口到 capability。
- `watchlist` 还只是 symbol 列表,没有成为真正的观察池领域对象。
- 总计划和长期目标规划还停留在文档,没有进入后端模型和一级页面。
- 写入事件与缓存失效规则仍分散在各调用点。
- 系统级验收矩阵不足。

## 3. 路线总览

| 阶段 | 名称 | 目标 | 完成标志 |
| :--- | :--- | :--- | :--- |
| M0 | 文档与入口基线 | 统一中文文档名、README 入口和开发顺序 | 新任务能从本文档找到下一步 |
| M1 | Provider 与能力边界硬化 | 让数据源只通过标准 capability 暴露能力 | AkShare 模式不会触发 TickFlow 专属路径 |
| M2 | 长期目标规划 MVP | 把"10 年 150 万"这类长期目标从 Markdown 变成可维护规划 | 能维护目标、情景、里程碑、持仓、净投入并看到 summary |
| M3 | 观察池领域模型 | 把自选股升级为有理由、状态和复盘记录的观察项 | 策略命中可进入观察池,观察项可被复盘 |
| M4 | 复盘闭环 | 复盘页读取总计划、观察池、策略结果形成次日计划 | 收盘后能生成明日观察和复盘记录 |
| M5 | 写入事件与缓存一致性 | 统一数据写入、缓存刷新和前端失效语义 | 写入路径都有事件声明和验收用例 |
| M6 | 系统级验收矩阵 | 覆盖 provider、pipeline、cache、SSE、降级行为 | 核心模式切换和失败场景可重复验证 |

## 4. M0: 文档与入口基线

目标:

- 所有产品和架构文档使用中文文件名。
- README 文档入口指向中文文档。
- 保留本开发路书作为后续任务的执行索引。

应做:

- 清理残留英文文档副本。
- 确认代码中读取的策略文档路径已经改为中文名。
- 后续新增设计文档优先用中文名。

验收:

- `docs/` 根目录没有产品文档英文名副本。
- README 能直达系统设计、开发路书、数据契约、能力契约、总计划设计和策略指南。
- `rg` 搜不到旧英文文档名引用,历史 Trellis 任务除外。

## 5. M1: Provider 与能力边界硬化

目标:

- TickFlow、AkShare 和未来 provider 都只作为数据源适配存在。
- 前端和业务服务只消费标准 capability,不猜 provider 私有细节。

应做:

1. 补充 capability limits。
   - 表达 rpm、batch、subscribe、min_interval、max_interval、max_history_days、realtime_allowed。
   - 不只把 `tier_label` 替换成另一个字符串。

2. 收口历史档位判断。
   - `backend/app/api/kline.py` 的分钟 K 长历史门控。
   - `backend/app/services/depth_service.py` 的轮询间隔范围。
   - `frontend/src/lib/capability-labels.tsx` 的 `tierRank()` UI 门控。
   - `frontend/src/components/data/DepthConfigCard.tsx` 的档位文案。

3. 建立 provider stage matrix。
   - 数据页和 pipeline 共享同一份阶段能力说明。
   - AkShare 不支持实时、分钟 K、五档、财务时要明确灰显或 403。

验收:

- `DATA_PROVIDER=akshare` 时不触发 TickFlow 能力探测和 TickFlow 专属网络路径。
- 免费数据源模式能完成关注范围日线同步、指标、选股、K 线、基础回测。
- TickFlow 模式行为不退化。

## 6. M2: 长期目标规划 MVP

目标:

- 将"10 年做到 150 万"这类长期目标规划变成系统里的一级对象。
- 不只是资产看板,还要承载目标设定、投入情景、年度里程碑、执行计划和年度复盘模板。
- 第一版只做规划、动态进度和静默状态,不做主动提醒和交易建议。

规划功能应覆盖:

- 目标设定: 初始本金、目标金额、目标日期、基准月投入、基准所需年化。
- 投入情景: 例如每月投入 1000 / 2000 / 3000 / 5000 时所需年化收益率。
- 年度里程碑: 第 1 年到第 10 年应达到的目标资产。
- 阶段计划: 第 1-3 年、第 4-6 年、第 7-10 年的执行重点。
- 风险规则: 不借钱、不加杠杆、单一标的上限、回撤处理规则。
- 年度复盘模板: 当前资产、本年投入、本年收益率、仓位结构、做对/做错、下一年重点。
- 当前行动清单: 根据目标差距和现金流给出静默的下一步事项,不弹提醒、不替用户下结论。

后端应做:

- 新增 `backend/app/services/investment_plan.py`。
- 新增 `backend/app/api/investment_plan.py`。
- 在 `backend/app/main.py` 注册 router。
- 使用 `data/user_data/` 下的 JSON / Parquet 文件保存:
  - `investment_plan.json`
  - `investment_plan_versions.jsonl`
  - `risk_profiles.json`
  - `plan_scenarios.json`
  - `plan_milestones.json`
  - `annual_review_templates.json`
  - `positions.parquet`
  - `contributions.parquet`
  - `position_adjustments.jsonl`
  - `portfolio_snapshots.parquet`

前端应做:

- 新增 `frontend/src/pages/InvestmentPlan.tsx`。
- 修改 `router.tsx`、`Layout.tsx`、`api.ts`、`queryKeys.ts`。
- 左侧导航新增"总计划"。

第一版页面包含:

- 顶部目标进度。
- 目标规划参数。
- 月投入情景表。
- 年度里程碑。
- 阶段执行计划。
- 资产拆解。
- 持仓结构。
- 持仓维护。
- 净投入记录。
- 年度复盘模板。
- 静默健康度。

验收:

- 空数据时页面可打开,返回默认草稿和空 summary。
- 用户能创建或载入"10 年做到 150 万"这类长期目标规划。
- 用户能调整目标金额、目标日期、初始本金和基准月投入。
- 用户能看到不同月投入对应的目标难度变化。
- 用户能看到年度里程碑和阶段执行计划。
- 用户能新增现金、股票、基金或其他资产。
- 用户能录入净投入和转出。
- 页面刷新后数据仍存在。
- summary 能计算计划资产、备用资产、总金融资产、累计净投入、简化投资收益和目标完成度。
- summary 能展示当前进度相对年度里程碑的领先/正常/落后状态。
- 用户能使用年度复盘模板记录一次年度复盘。
- 编辑持仓时写入轻量调整日志。

## 7. M3: 观察池领域模型

目标:

- 把普通 watchlist 升级为观察池,成为复盘、策略和监控共同依赖的领域对象。

建议模型:

```text
ObservationItem
  id
  symbol
  name
  source
  source_ref
  thesis
  trigger_price
  invalid_price
  target_price
  status
  priority
  tags
  intended_bucket
  planned_capital
  risk_note
  created_at
  updated_at
  review_notes[]
```

应做:

- 后端保留旧 watchlist API 兼容层。
- 新增观察项增删改查和状态变更。
- 策略命中结果支持一键加入观察池。
- 监控规则可以引用观察项的触发价和失效价。

验收:

- 用户能记录"为什么关注、什么条件触发、什么条件失效"。
- 策略命中可进入观察池。
- 观察项可归档、失效、触发并写复盘备注。
- 旧自选数据可迁移或兼容展示。

## 8. M4: 复盘闭环

目标:

- 让复盘页从行情看板升级为次日计划生成入口。

应做:

- Dashboard 读取总计划 summary。
- Dashboard 读取观察池和策略结果。
- 复盘页生成"明日观察清单"和"需要复盘的观察项"。
- 记录复盘备注,沉淀到观察项或日度复盘记录。

验收:

- 收盘后同步数据后,用户能看到市场状态、策略表现、观察池状态和总计划摘要。
- 用户能从复盘页形成明日观察池。
- 次日结果能回写到观察项复盘记录。

## 9. M5: 写入事件与缓存一致性

目标:

- 减少"数据写了但页面没刷新"和"缓存旧数据误导"。

建议新增:

```text
DataMutation
  event
  tables
  duckdb_views
  polars_caches
  frontend_queries
  sse_events
```

应做:

- 先文档化事件表,再实现轻量 registry。
- pipeline、quote service、ext data、monitor alert 都上报同一种事件。
- 前端 query invalidation 与后端事件语义对齐。
- job result 明确写入哪些表、哪些 scope、哪些失败。

验收:

- 每个写入路径都能说清楚刷新和失效范围。
- 刷新、切页、重连后的页面状态一致。
- AkShare 同步失败保留旧数据,并能在 job result 中看到失败原因。

## 10. M6: 系统级验收矩阵

目标:

- 从功能可用走向系统可信。

至少覆盖:

- TickFlow none/free/starter+ capability 行为。
- AkShare 手动同步、失败保留旧数据、关注范围同步、全市场同步。
- pipeline job 重入、进度、失败结果。
- enriched cache 刷新和前端 SSE invalidation。
- 回测数据边界。
- 扩展数据 schema 发现、写入和 join。
- 总计划空数据、持仓估值、净投入、版本历史。
- 观察池迁移、策略加入观察池、触发/失效状态。

验收:

- 关键模式有可重复执行的测试或手工验收脚本。
- 新功能 PR 必须说明它影响哪些验收项。

## 11. 当前建议的下一步

当前最建议执行顺序:

1. 完成 M0: 清理残留英文文档副本。
2. 做 M1 的最小闭环: capability limits + 清理真实门控里的 `tier_label` / `tierRank`。
3. 开始 M2: 长期目标规划后端最小闭环。
4. 再做 M2 前端一级页面,先把"10 年做到 150 万"规划载入系统。
5. 长期目标规划稳定后再进入 M3 观察池模型。

不要优先做:

- 新增更多行情展示页。
- 自动交易或券商接入。
- 完整家庭资产负债表。
- 复杂收益归因。
- 高频实时能力。

这些能力会扩大边界,应在总计划和观察池闭环稳定后再评估。

## 12. 新任务开工检查

每个新开发任务开始前先回答:

1. 属于 M1 到 M6 的哪一阶段?
2. 服务哪条主工作流:长期规划、总计划、复盘、观察池、策略、监控、回测、数据?
3. 依赖哪些 capability? 不满足时怎么降级?
4. 读写哪些本地表或用户数据文件?
5. 写入后刷新哪些 DuckDB view、Polars cache、前端 query?
6. 是否影响 AkShare / TickFlow 模式切换?
7. 需要补哪些验收用例?

答不上来时,先补设计再写代码。
